Registro Fotográfico de Obra — Enhancement Specification v1.1

1. Objective

Improve the existing Android application with:

1. More flexible mass structure creation.
2. Inline service creation during a check-in.
3. Configurable PDF report appearance and organization.
4. Corrections to mass-generation numbering.
5. Cloning of complete construction structures.

All features must continue to work completely offline.

All user-facing text must remain in Brazilian Portuguese.

---

2. Mass creation improvements

2.1 Flexible item-name composition

The mass creation tool must allow the user to configure how generated item names are composed.

Add these fields:

- Prefixo
- Sufixo
- Formato do número
- Adicionar zero à esquerda
- Quantidade mínima de dígitos

The final generated name must follow:

{prefixo} {número formatado} {sufixo}

Blank values must not produce unnecessary spaces.

Examples

Configuration:

Prefixo: Pavimento
Número: 1
Sufixo: vazio

Result:

Pavimento 1

Configuration:

Prefixo: vazio
Número: 1
Sufixo: Pavimento

Result:

1 Pavimento

Configuration:

Prefixo: Torre
Número: 1
Sufixo: Residencial

Result:

Torre 1 Residencial

The application must automatically normalize spaces between the prefix, number and suffix.

The user must not need to manually enter a trailing or leading space in the prefix or suffix fields.

2.3 Ordinal behavior with zero-padding

Ordinal formatting and zero-padding must work together.

Examples with two digits:

01º Pavimento
02º Pavimento
03º Pavimento

Examples with three digits:

001
002
003

The number-processing order must be:

1. Generate the numeric value.
2. Apply zero-padding.
3. Add the ordinal symbol when selected.
4. Compose the prefix and suffix.

Example:

Value: 1
Minimum digits: 2
Ordinal masculine: enabled
Prefix: Pavimento

Result:

Pavimento 01º

---

2.4 Independent zero-padding configuration

The option to add leading zeros must be independent for every generated entity.

For example, the user may configure:

Pavimentos:
01º Pavimento
02º Pavimento
03º Pavimento

while configuring units as:

Apartamento 101
Apartamento 102
Apartamento 103

The floor zero-padding configuration must not modify the unit numbering configuration.

Likewise, unit zero-padding must not modify floor numbering.

Each generated entity configuration must maintain its own values:

FloorGenerationSettings
- useLeadingZeros
- minimumDigits
- numberFormat
- prefix
- suffix

UnitGenerationSettings
- useLeadingZeros
- minimumDigits
- numberFormat
- prefix
- suffix

Do not reuse one shared state variable for both floor and unit formatting.

3. Support generation starting from zero

3.1 Functional requirement

Zero must be considered a valid starting and ending number.

Example:

De: 0
Até: 3

Expected result:

Pavimento 0
Pavimento 1
Pavimento 2
Pavimento 3

The interval is inclusive.

Starting from zero must generate the floor numbered zero.

---

3.2 Validation rules

Valid examples:

0 to 0
0 to 10
1 to 10
10 to 10

Invalid examples:

- Empty starting number.
- Empty ending number.
- Starting number greater than ending number, unless descending generation is explicitly supported.
- Non-integer values.

Do not treat zero as an empty or missing value.

---

3.3 Likely implementation issue

Avoid validation or defaulting patterns such as:

if (!startNumber) {
    // invalid
}

or equivalent logic where zero is interpreted as false.

Also avoid logic equivalent to:

startNumber = providedValue OR 1

because zero would incorrectly become one.

Use explicit null validation:

if (startNumber == null) {
    // invalid
}

The database and domain model must accept zero as a valid numeric reference.

4. Add a service during a check-in

4.1 Objective

Allow the engineer to create a missing Serviço without leaving the current check-in or losing the selected location.

Current hierarchy context must remain preserved:

Quadra
→ Prédio
→ Pavimento
→ Unidade
→ Serviço

---

4.2 Service selection screen

At the bottom or top of the Serviço selection screen, add a visible action:

«Adicionar novo serviço»

Use both an icon and text.

Do not hide this action only inside an overflow menu.

Suggested layout:

Serviços

[ Hidráulica                 ]
[ Elétrica                   ]
[ Pintura                    ]

[ + Adicionar novo serviço  ]

---

4.3 Creation flow

When the user selects Adicionar novo serviço, open a modal dialog or bottom sheet.

Fields:

Nome do serviço

Actions:

- Cancelar
- Adicionar

Example:

Adicionar novo serviço

Nome do serviço
[ Instalação de portas       ]

[Cancelar] [Adicionar]

---

4.4 Behavior after creation

After successfully creating the service:

1. Add it to the Obra-level service list.
2. Make it immediately available in all future units.
3. Keep the current check-in active.
4. Preserve the selected Quadra, Prédio, Pavimento and Unidade.
5. Automatically select the newly created service.
6. Open the photo capture screen for that service.

The user must not be returned to the home screen or structure configuration screen.

---

4.5 Duplicate validation

Service names must be compared after:

- Trimming spaces.
- Ignoring letter case.
- Normalizing repeated spaces.

These values must be considered duplicates:

Hidráulica
hidráulica
 HIDRÁULICA
Hidráulica   

When a duplicate exists, show:

«Já existe um serviço com este nome.»

Optionally provide:

«Usar serviço existente»

Selecting it should select the existing service and continue to the camera.

---

4.6 Empty and invalid names

Do not allow:

- Empty service names.
- Names containing only spaces.
- Names longer than the configured safe maximum.

Recommended maximum:

80 characters

6.4 Grouping rules

The grouping setting changes the PDF section order only.

It must not change:

- Photo associations.
- Stored hierarchy.
- Check-in data.
- Export folder structure.
- Database relationships.

Exported image folders must continue following the physical hierarchy:

Quadra
→ Prédio
→ Pavimento
→ Unidade
→ Serviço

The report grouping configuration must not modify ZIP organization.

---

6.5 Sorting inside groups

Each grouping field must use its configured "sortOrder".

Photos inside the final group must be sorted by:

capturedAt ascending

The sorting must remain deterministic.

7. Report color configuration

7.1 Color setting

Add a setting named:

«Cor principal do relatório»

The selected color replaces the current default blue in:

- Report header.
- Section titles.
- Divider lines.
- Page-number accents.
- Relevant table or label details.

Do not recolor the construction photos.

---

7.2 Color selection interface

Provide:

- A small set of predefined colors.
- A custom color picker.
- Optional hexadecimal input.

Suggested predefined options:

- Azul
- Verde
- Laranja
- Vermelho
- Cinza
- Preto

The selected color must display a live preview.

---

7.3 Contrast

The PDF renderer must automatically choose readable foreground text.

For dark selected colors:

Use white text.

For light selected colors:

Use dark text.

Do not always use white text regardless of the selected background color.

Validate contrast before generating the report.

---

7.4 Invalid or transparent colors

Do not allow:

- Fully transparent colors.
- Invalid hexadecimal values.
- Colors that make content unreadable without contrast correction.

Persist the final color as an ARGB or RGB value in local settings.

1. Copy it into the application’s local storage.

4. Preserve its aspect ratio.
5. Store its reference in local settings.

- Maximum source file size: 5 MB.

- Reject unsupported formats.
- Resize very large images before PDF use.
- Never stretch the logo.

«Não foi possível utilizar esta imagem como logo.»

8.4 Logo placement

Requirements:

- Preserve aspect ratio.
- Use a maximum width and height.
- Do not let the logo consume excessive page space.
- Do not distort portrait or horizontal logos.
- When no logo exists, use the full header space for report information.

- Generate first, then apply page numbering in a second pass.

16. Clone confirmation and progress

16.1 Confirmation

Before cloning a floor, building or block, display:

- Origin.
- Destination.
- New item name.
- Number of child records to create.
- Clear statement that photos will not be copied.

Example:

«A estrutura será duplicada sem fotos ou registros de vistoria.»

---

16.2 Transaction

Every cloning operation must run inside a database transaction.

If any part fails:

- Roll back the complete operation.
- Do not leave a partially cloned structure.
- Preserve the original structure unchanged.

---

16.3 Progress

For large Block or Building clones, show progress.

Example:

Duplicando estrutura...

3 de 4 prédios processados

Do not freeze the interface.

Prevent the user from starting the same cloning operation repeatedly while it is already running.

---

16.4 New identifiers

Every cloned entity must receive a new unique ID.

Do not reuse identifiers from the source structure.

Maintain an internal map during the operation:

oldBlockId -> newBlockId
oldBuildingId -> newBuildingId
oldFloorId -> newFloorId
oldUnitId -> newUnitId

This ensures that cloned children reference the new cloned parents.

---

16.5 Name collisions

The new top-level cloned item must have a unique name inside its destination parent.

Examples:

- Unit name unique inside the destination floor.
- Floor name unique inside the destination building.
- Building name unique inside the destination block.
- Block name unique inside the project.

If the name already exists, show:

«Já existe um item com este nome no destino selecionado.»

Do not silently add automatic suffixes such as "(1)" unless the user explicitly chooses that name.

---

16.6 Archived items

By default, do not clone archived descendants.

Example:

- A cloned building includes active floors and units.
- Archived units remain only in the original building.

This avoids reintroducing structures that the user intentionally removed from active use.

18. Acceptance criteria

18.1 Prefix and suffix

Given:

Prefix: Pavimento
Start: 1
End: 3

Generate:

Pavimento 1
Pavimento 2
Pavimento 3

Given:

Suffix: Pavimento
Start: 1
End: 3

Generate:

1 Pavimento
2 Pavimento
3 Pavimento

---

18.2 Ordinal numbers

Given:

Prefix: Pavimento
Number format: Ordinal masculine
Start: 1
End: 3

Generate:

Pavimento 1º
Pavimento 2º
Pavimento 3º

---

18.3 Independent padding

Given:

Floors:
Leading zeros: enabled
Digits: 2

Units:
Leading zeros: disabled

Floor names must contain:

01
02
03

Unit names must remain unpadded according to their own pattern.

Changing the floor setting must not change the unit setting.

---

18.4 Floor zero

Given:

Start: 0
End: 2

The application must create three floors:

0
1
2

---

18.5 Inline service creation

Given the user is at:

Quadra A
Prédio 1
2º Pavimento
Apartamento 204

When the user creates:

Instalação de portas

the application must:

- Save the new service globally for the Obra.
- Preserve the current hierarchy.
- Select the new service.
- Open the camera.
- Keep the current check-in active.

---

18.6 Report grouping

Given the grouping order:

Serviço
Prédio
Pavimento
Unidade

The PDF must organize its sections in exactly that order.

The underlying stored photo hierarchy must remain unchanged.

---

18.7 Color

Given the user selects green:

- PDF section elements use green instead of default blue.
- Text remains readable.
- Photos remain unchanged.

---

18.8 Logo

Given the user selects a valid PNG logo:

- The logo is copied locally.
- It remains available offline.
- It appears in the report header.
- Its aspect ratio is preserved.

After the user removes the logo:

- New reports no longer display it.
- Existing exported reports remain unchanged.

---

18.9 Pagination

Given pagination is disabled:

- No page number appears.

Given current page only:

1
2
3

Given current and total:

1/3
2/3
3/3

---

18.10 Clone unit

Duplicating a unit must:

- Create a new unit ID.
- Preserve the selected unit type.
- Create no photos.
- Create no history.
- Require a unique destination name.

---

18.11 Clone floor

Duplicating a floor containing six active units must:

- Create one new floor.
- Create six new units.
- Give every entity a new ID.
- Copy no photos or history.

---

18.12 Clone building

Duplicating a building containing 23 floors and 138 units must:

- Create one new building.
- Create 23 new floors.
- Create 138 new units.
- Preserve ordering.
- Copy no photographic data.

---

18.13 Clone block

Duplicating a block must recreate its complete active structural tree using new IDs.

No photos, sessions, reports or historical records may be connected to the duplicated structure.

---

19. Implementation priority

Recommended order:

1. Fix generation starting at zero.
2. Separate leading-zero settings.
3. Add prefix, suffix and ordinal formatting.
4. Add inline service creation.
5. Add unit, floor, building and block cloning.
6. Add report color and pagination.
7. Add report logo.
8. Add configurable ordered report grouping.

The zero-generation and shared-zero-padding issues are correctness bugs and should be handled before adding new functionality.
